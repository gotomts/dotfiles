# 実装ドキュメント - 各ノートのテンプレート

実装フェーズで作ると役立つドキュメントのテンプレート。すべてのサービスが全部必要なわけではない。サービスの複雑さに応じて取捨選択する。

## 推奨される作成タイミング

| ノート | 作成タイミング | 必要度 |
|---|---|---|
| 機能一覧 | コア共通骨格が終わった後、実装着手前 | 中-高 |
| パス設計 | Web アプリの場合、実装着手前 | 中-高 |
| モジュール一覧 | アーキテクチャが複雑な場合、実装着手前 | 中 |
| ドメインモデリング | ドメインが複雑な場合（DDD 採用時） | 中 |
| ユビキタス言語 | ドメイン用語が多い場合 | 中 |
| ワイヤーフレーム | UI が重要な場合 | サービスによる |
| CLAUDE.md | Claude Code を使う場合 | 高（Claude Code 利用時） |

## 機能一覧

### 目的

実装すべき機能を網羅的にリストアップする。**企画専任のドキュメント**であり、ここでは機能の存在と最低限の補足（分岐・制約・重要な仕様）までを書く。

このページは後段の **分解スキル**（機能一覧を入力として、受入条件の生成・依存順付け・実装単位 issue 化を行う別スキル）の入力となる前提で書く。分解スキルが機械的に読み取れるよう、以下の規約を必ず守る。

### 聞くこと

- 各画面・各エンドポイントで提供する機能
- 機能が属するモジュール（モジュール一覧と命名を合わせる）
- 機能ごとの Phase（Phase 1 / Phase 2 / 将来候補）
- 機能内の分岐・条件・重要な仕様（補足見出しで書く対象）

**受入条件・完了条件・テストケースは聞かない**。設計セッションでの企画の発散を妨げないため、ここでは「種」までで止める。

### 規約

#### 1. ID 体系：接頭辞方式

形式：`F-{MODULE}-{連番}`

- `MODULE` は大文字英字（短縮形可、例：AUTH, FEED, PROFILE）
- 連番は 2 桁ゼロ埋め（01, 02, ..., 99）
- モジュール名は「モジュール一覧」または対応する設計ページのモジュール命名と一致させる

例:

- `F-AUTH-01` GitHub アカウントでサインアップ
- `F-FEED-01` 最新タブの記事フィード閲覧
- `F-DETAIL-01` 記事の日本語要約閲覧

モジュール構造と機能 ID を 1 対 1 対応させると、分解スキルが依存関係を ID から推論しやすく、Linear などのタスク管理ツールでも所属がひと目でわかる。

#### 2. ページ構造

- `# 機能一覧` は 1 階層（h1）
- **Phase の区別は `##` 見出し**（`## Phase 1`, `## Phase 2`, `## 将来候補`）
- **モジュール（カテゴリ）の区別は `###` 見出し**
- 表は `| ID | 機能 |` の 2 列固定

#### 3. 補足見出し

機能の中に**分岐・条件・複雑な仕様**がある場合、その機能の補足を ID 付き見出しで書く。

形式：`#### F-{MODULE}-{連番} タイトル`

- 補足見出しは **`####`（4 階層）固定**
- 見出しの形は **「ID + 半角スペース + タイトル」**
- 補足は対応する表と同じ Phase / モジュールの **直下** に置く
- 補足はあくまで「種」（分岐・制約・重要な仕様）。**受入条件は書かない**

#### 4. 受入条件は書かない

このテンプレートに「受入条件」「完了条件」「テストケース」などのセクションは設けない。

理由：受入条件の生成は後段の分解スキルの責務。設計フェーズでこれらを書き始めると、企画の発散と実装計画が混ざって意思決定の粒度が崩れる。

ただし、受入条件を書かない代わりに、**機能の意図や制約は補足見出しで書く**こと。例：「F-AUTH-01 はソーシャル認証のみ（メール認証は提供しない）」のような決定事項は補足に書く。

### テンプレート（実例：anttt 想定）

````markdown
# 機能一覧

## このページの位置づけ

サービスで提供する機能を Phase × モジュール単位で網羅する企画ドキュメント。各機能には `F-{MODULE}-{連番}` の ID を振り、分岐・制約・重要な仕様は補足見出しで書く。受入条件は書かない（後段の分解スキルが生成）。

参照元: 機能スコープ、モジュール一覧。

## Phase 1

### auth モジュール

| ID | 機能 |
| --- | --- |
| F-AUTH-01 | GitHub アカウントでサインアップ |
| F-AUTH-02 | Google アカウントでサインアップ |
| F-AUTH-03 | アカウント削除 |

#### F-AUTH-01 ソーシャル認証のみ

- メール / パスワード認証は提供しない（運用負荷とアカウント乗っ取り対応を回避）
- 「Sign in with」ボタンは GitHub / Google の 2 種類のみ表示
- 将来 Apple Sign-In を追加する想定だが Phase 1 では対象外（F-AUTH-10 で扱う）

### feed モジュール

| ID | 機能 |
| --- | --- |
| F-FEED-01 | 最新タブの記事フィード閲覧 |
| F-FEED-02 | お気に入りタブの記事フィード閲覧 |

### detail モジュール

| ID | 機能 |
| --- | --- |
| F-DETAIL-01 | 記事の日本語要約閲覧 |
| F-DETAIL-02 | 「日本語で表示」ボタン |

#### F-DETAIL-02 「日本語で表示」ボタンの 4 分岐

- 未ログイン → サインインモーダル
- 拡張機能未インストール → インストール案内モーダル
- Free でトークン残量不足 → Pro 案内モーダル
- すべて OK → 原文サイトに遷移、拡張機能が翻訳実行

## Phase 2

### auth モジュール

| ID | 機能 |
| --- | --- |
| F-AUTH-10 | Apple Sign-In |

### feed モジュール

| ID | 機能 |
| --- | --- |
| F-FEED-10 | タグでフィードを絞り込み |

## 将来候補

### detail モジュール

| ID | 機能 |
| --- | --- |
| F-DETAIL-90 | 要約の音声読み上げ |
````

## パス設計

### 目的
URL ルーティングを設計する。RESTful な設計を意識しつつ、ユーザーにとって直感的なパスを決める。

### 聞くこと
- 主要な画面の URL
- API エンドポイント
- 管理画面の URL（別ドメイン or サブパス）

### テンプレート

```markdown
# パス設計

## Web ページ

| パス | 内容 | 認証要否 |
|---|---|---|
| `/` | トップ | No |
| `/login` | ログイン | No |
| `/dashboard` | ダッシュボード | Yes |

## API エンドポイント

| メソッド | パス | 内容 | 認証要否 |
|---|---|---|---|
| GET | `/api/users/me` | 自分の情報取得 | Yes |
| POST | `/api/posts` | 投稿作成 | Yes |

## 管理画面

| パス | 内容 |
|---|---|
| `/admin` | 管理画面トップ |
| `/admin/users` | ユーザー一覧 |

## 命名規則

- リソース名: 複数形 / 小文字 / kebab-case
- ID は数値 / ULID / UUID のどれ?
```

## ドメインモデリング

### 目的
ビジネスドメインのエンティティと関連を整理する。DDD を採用する場合は特に重要。

### 聞くこと
- 主要なエンティティ
- 各エンティティの属性
- エンティティ間の関連
- 集約の境界

### テンプレート

```markdown
# ドメインモデリング

## エンティティ一覧

### User
- 属性: id, email, name, createdAt
- 関連: 1人の User は複数の Post を持つ
- 集約: User 集約のルート

### Post
- 属性: id, title, content, userId, status, publishedAt
- 関連: 1つの Post は 1人の User に属する
- ステータス: draft, published, archived

### Comment
- 属性: id, postId, userId, content, createdAt
- 関連: Comment は Post と User に属する

## ステート遷移

### Post.status
```
draft → published → archived
draft → archived
```

## 集約

- User 集約: User
- Post 集約: Post, Comment
- それぞれのトランザクション境界

## リポジトリ

- UserRepository
- PostRepository
- CommentRepository
```

## ユビキタス言語

### 目的
ドメイン特有の用語を整理する。チーム内・コード内・ユーザー向けの表現を統一する。

### 聞くこと
- そのサービス固有の用語
- 用語の定義
- 別名・避けるべき表現

### テンプレート

```markdown
# ユビキタス言語

## ドメイン用語

| 用語 | 定義 | コード上の名称 | UI 上の表記 |
|---|---|---|---|
| ユーザー | サービス利用者 | User | ユーザー |
| 投稿 | ユーザーが作成したコンテンツ | Post | 投稿 |

## 避けるべき用語

| 避ける | 代わりに使う | 理由 |
|---|---|---|
| メンバー | ユーザー | 「メンバー」は有料プランの含意を持つため |

## 状態の用語

| 状態 | 意味 | コード上 | UI 上 |
|---|---|---|---|
| 下書き | 公開前 | draft | 下書き |
| 公開済み | 公開後 | published | 公開済み |
```

## 実装ドキュメントを作るときの注意

### 完璧を目指さない

実装ドキュメントは「作りながら更新する」もの。最初から完璧を目指さず、骨格を作って実装と並行して育てる。

### 重複を避ける

機能スコープ → 機能一覧、アーキテクチャ → モジュール一覧 のように、粒度の異なる重複ドキュメントがある。最新の情報がどこにあるか明確にする（より詳細な方を優先）。

### ワイヤーフレームについて

UI が重要なサービスの場合、ワイヤーフレームを HTML プロトタイプとして作る。詳細は `references/wireframe-workflow.md` を参照。
