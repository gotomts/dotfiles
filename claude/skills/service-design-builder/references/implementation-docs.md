# 実装ドキュメント - 各ページのテンプレート

実装フェーズで作ると役立つドキュメントのテンプレート。すべてのサービスが全部必要なわけではない。サービスの複雑さに応じて取捨選択する。

## 推奨される作成タイミング

| ページ | 作成タイミング | 必要度 |
|---|---|---|
| 機能一覧 | コア共通骨格が終わった後、実装着手前 | 中-高 |
| パス設計 | Web アプリの場合、実装着手前 | 中-高 |
| モジュール一覧 | アーキテクチャが複雑な場合、実装着手前 | 中 |
| ドメインモデリング | ドメインが複雑な場合（DDD 採用時） | 中 |
| ユビキタス言語 | ドメイン用語が多い場合 | 中 |
| ワイヤーフレーム | UI が重要な場合 | サービスによる |
| CLAUDE.md | Claude Code を使う場合 | 高（Claude Code 利用時） |

## 機能一覧

### 目的
実装すべき機能を網羅的にリストアップする。機能スコープページよりも詳細で、実装単位に近い粒度。

### 聞くこと
- 各画面・各エンドポイントで提供する機能
- 機能ごとの優先度
- 機能ごとの依存関係

### テンプレート

```markdown
# 機能一覧

## ユーザー向け機能

### 認証
- [ ] サインイン
- [ ] サインアップ
- [ ] パスワードリセット
- [ ] アカウント削除

### コア機能
- [ ] 機能1
- [ ] 機能2

### 設定
- [ ] プロフィール編集
- [ ] 通知設定

## 管理者向け機能

- [ ] ユーザー管理
- [ ] コンテンツ管理

## バッチ・非同期処理

- [ ] バッチ1
- [ ] ジョブ2

## 優先度

- P0 (必須): 
- P1 (重要): 
- P2 (あれば良い): 
```

## パス設計

### 目的
URL ルーティングを設計する。RESTful な設計を意識しつつ、ユーザーにとって直感的なパスを決める。

### 聞くこと
- 主要な画面の URL
- API エンドポイント
- 管理画面の URL（別ドメイン or サブパス）

### テンプレート

```markdown
# パス設計

## Web ページ

| パス | 内容 | 認証要否 |
|---|---|---|
| `/` | トップ | No |
| `/login` | ログイン | No |
| `/dashboard` | ダッシュボード | Yes |

## API エンドポイント

| メソッド | パス | 内容 | 認証要否 |
|---|---|---|---|
| GET | `/api/users/me` | 自分の情報取得 | Yes |
| POST | `/api/posts` | 投稿作成 | Yes |

## 管理画面

| パス | 内容 |
|---|---|
| `/admin` | 管理画面トップ |
| `/admin/users` | ユーザー一覧 |

## 命名規則

- リソース名: 複数形 / 小文字 / kebab-case
- ID は数値 / ULID / UUID のどれ?
```

## ドメインモデリング

### 目的
ビジネスドメインのエンティティと関連を整理する。DDD を採用する場合は特に重要。

### 聞くこと
- 主要なエンティティ
- 各エンティティの属性
- エンティティ間の関連
- 集約の境界

### テンプレート

```markdown
# ドメインモデリング

## エンティティ一覧

### User
- 属性: id, email, name, createdAt
- 関連: 1人の User は複数の Post を持つ
- 集約: User 集約のルート

### Post
- 属性: id, title, content, userId, status, publishedAt
- 関連: 1つの Post は 1人の User に属する
- ステータス: draft, published, archived

### Comment
- 属性: id, postId, userId, content, createdAt
- 関連: Comment は Post と User に属する

## ステート遷移

### Post.status
```
draft → published → archived
draft → archived
```

## 集約

- User 集約: User
- Post 集約: Post, Comment
- それぞれのトランザクション境界

## リポジトリ

- UserRepository
- PostRepository
- CommentRepository
```

## ユビキタス言語

### 目的
ドメイン特有の用語を整理する。チーム内・コード内・ユーザー向けの表現を統一する。

### 聞くこと
- そのサービス固有の用語
- 用語の定義
- 別名・避けるべき表現

### テンプレート

```markdown
# ユビキタス言語

## ドメイン用語

| 用語 | 定義 | コード上の名称 | UI 上の表記 |
|---|---|---|---|
| ユーザー | サービス利用者 | User | ユーザー |
| 投稿 | ユーザーが作成したコンテンツ | Post | 投稿 |

## 避けるべき用語

| 避ける | 代わりに使う | 理由 |
|---|---|---|
| メンバー | ユーザー | 「メンバー」は有料プランの含意を持つため |

## 状態の用語

| 状態 | 意味 | コード上 | UI 上 |
|---|---|---|---|
| 下書き | 公開前 | draft | 下書き |
| 公開済み | 公開後 | published | 公開済み |
```

## 実装ドキュメントを作るときの注意

### 完璧を目指さない

実装ドキュメントは「作りながら更新する」もの。最初から完璧を目指さず、骨格を作って実装と並行して育てる。

### 重複を避ける

機能スコープ → 機能一覧、アーキテクチャ → モジュール一覧 のように、粒度の異なる重複ドキュメントがある。最新の情報がどこにあるか明確にする（より詳細な方を優先）。

### ワイヤーフレームについて

UI が重要なサービスの場合、ワイヤーフレームを HTML プロトタイプとして作る。詳細は `references/wireframe-workflow.md` を参照。
