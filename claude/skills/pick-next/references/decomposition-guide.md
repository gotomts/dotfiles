# decomposition-guide

`pick-next` の Step 6B（新規テーマ → spec/plan 書き出し）で sub-issue を分解するときの指針。

## 階層は 1 段固定

親 Issue + sub-issue のみ。**sub-issue の sub-issue は作らない**。`create-issue` の現状仕様と整合する。

## 粒度の目安

1 sub-issue = **半日〜2 日 / PR 1 本**。

## 判断ロジック

選定したテーマの **コスト軸スコア**（`score-axes.md` 参照）で分岐:

| コスト | sub-issue 件数 | 補足 |
|--------|--------------|------|
| 小 (1d 以下) | 0 件（親 Issue だけ） | 「これを 2 つ以上に切る価値はないですね、親だけにします」と提案 |
| 中 (2-5d) | 2〜4 件 | 標準的な分解 |
| 大 (1w+) | エスカレーション提案 | 「このスキルの守備範囲を超えてる気配。`superpowers:brainstorming` 本体に切り替えますか？」と確認 |

ユーザーが「コスト大でも `pick-next` で進めたい」と返した場合は、件数を 5〜8 件まで許容する。それでも収まらないなら強制的にエスカレーション。

## 各 sub-issue に必要な情報

最低限以下を埋める:

- **タイトル**（25 文字以内、imperative）
- **概要**（1〜3 行）
- **変更対象ファイル**（推測でも書く）
- **受入条件**（チェックリスト 2〜5 項目、テストを含む）
- **依存関係**（前ステップ完了が必要なら明記）

## 依存関係の扱い

plan に「Step B は Step A の後」と書かれていれば、`create-issue` 側で `blocks` / `blocked by` として登録される。記述例:

```
**依存**: Step 1 完了後
```

または:

```
**依存**: なし（並列実行可）
```

## 1 件しか出ない場合の提案

Step 6B でユーザーと分解を進めた結果、sub-issue が 1 件にしかならなかった場合:

> 「sub-issue が 1 件しか出ませんでした。親 Issue だけで十分では？」

ユーザーが OK したら sub-issue を作らず、親 Issue 1 本だけで `create-issue` を呼ぶ。plan も「ステップ 1: <作業全体>」と書くだけで OK。
