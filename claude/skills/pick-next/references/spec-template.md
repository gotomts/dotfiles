# spec-template

`pick-next` の Step 6B で `docs/superpowers/specs/<date>-<slug>-design.md` に書き出す軽量 spec のテンプレ。

## ファイル名規則

- 日付: `date +%Y-%m-%d`（書き出し時点の今日の日付）
- slug: タイトルから ASCII 化した kebab-case（例: `auth-rate-limit-improve`）
- 拡張子: `-design.md`

例: `docs/superpowers/specs/2026-05-14-auth-rate-limit-improve-design.md`

## テンプレ全文

````markdown
---
labels: [feature]
pick-next: true
---

# <タイトル>

## 目的・背景

<5〜10 行。「なぜ今これか」「何の問題を解くか」を明記。
3 軸スコアの根拠もここに含める。>

## 採用判断（3 軸スコア）

- インパクト: 高 (3) — <理由>
- モチベーション: 高 (3) — <理由>
- コスト: 中 (2-5d) — <内訳の概算>

## 受入条件

- [ ] <ユーザー視点で完了が確認できる条件>
- [ ] <技術的な条件>
- [ ] <テストが追加されている>

## スコープ外

- <今回はやらないこと>
- <将来の拡張ポイント>

## 却下した代替案

| 候補 | インパクト | モチベ | コスト | 却下理由 |
|------|-----------|--------|--------|----------|
| A: ... | 高 | 中 | 中 | <理由> |
| C: ... | 高 | 低 | 中 | <理由> |

## アプローチ（任意、必要なら）

<技術的な方針が複数ある場合、選んだ方針と理由>
````

## フィールドの意味

- **labels**: `create-issue` がそのまま Issue ラベルに転用する。デフォルト `[feature]`
- **pick-next: true**: 中断再開検出と `--history` 一覧で識別子として使う。**必須**
- **目的・背景**: `create-issue` が親 Issue 本文の「## 目的・背景」に転載
- **採用判断（3 軸スコア）**: `pick-next` 固有の追加情報。`create-issue` は本文の補足として転載
- **受入条件**: `create-issue` が親 Issue 本文の「## 受入条件」に転載
- **スコープ外**: 親 Issue 本文に転載
- **却下した代替案**: `pick-next` 固有。`create-issue` は本文末尾に転載
- **アプローチ**: あれば `create-issue` が「設計コメント」として親 Issue にコメント追加

## 記入例

````markdown
---
labels: [feature, security]
pick-next: true
---

# auth-rate-limit-improve: ログイン API のレート制限を強化

## 目的・背景

現状 `/api/auth/login` には IP ベースのレート制限が無く、ブルート
フォース攻撃に対して脆弱。直近のログ監査で同一 IP からの大量試行
を確認しており、優先度を上げて対応する。

## 採用判断（3 軸スコア）

- インパクト: 高 (3) — セキュリティリスクの直接緩和
- モチベーション: 中 (2) — やる必要は感じるが心は躍らない
- コスト: 中 (3d) — middleware 追加 + テスト + 本番ロールアウト

## 受入条件

- [ ] 同一 IP からの login 試行が 1 分間に 5 回を超えると 429 を返す
- [ ] レート制限カウンタは Redis に保存される（in-memory 不可）
- [ ] 統合テストで 429 動作を確認できる
- [ ] エラーレスポンスに retry-after ヘッダが含まれる

## スコープ外

- ログイン以外のエンドポイントのレート制限（次回テーマ）
- captcha 統合（コスト過大、別 PR）

## 却下した代替案

| 候補 | インパクト | モチベ | コスト | 却下理由 |
|------|-----------|--------|--------|----------|
| プロフィール画面の刷新 | 中 | 高 | 中 | 楽しいがリスク低、後回しでよい |
| SEO メタタグ最適化 | 高 | 低 | 小 | モチベ低、心が乗らないので先送り |

## アプローチ

express-rate-limit + Redis ストア。`@nestjs/throttler` も検討した
が、middleware 単体で済むほうが依存が軽いため express-rate-limit
を採用。
````
