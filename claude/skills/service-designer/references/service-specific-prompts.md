# サービスタイプ別の固有論点ヒント

サービスのタイプに応じて、固有論点として検討すべきものをまとめる。
このリストは「これを必ず作れ」ではなく「対話の中でこういう論点が出てきたら察知してノート化を提案する」ためのヒント。

## 共通の察知サイン

以下のような兆候があったらサービス固有論点としてノート化を検討:

- ユーザーが特定の論点について何度も話す
- 1 つの論点で複数の選択肢を比較している
- 数値（料金、頻度、サイズ、品質基準など）を決めている
- 「これを変更したら他のノートにも影響する」と感じる根幹的な判断
- ドメイン特有の用語が頻出する

## コンテンツ集約・配信サービス（例: anttt）

主にコンテンツを集めて加工して配信するサービス。RSS リーダー、ニュースアプリ、まとめサイト系。

### 検討すべき固有論点
- **データソース管理**: 何を、どこから、どう取得するか
- **カテゴリ・タグ設計**: 分類体系の設計
- **コンテンツ品質基準**: 文字数、口調、文体
- **重複検知の扱い**: 複数ソースから来た同じ内容の扱い
- **更新頻度・コスト管理**: クロール頻度と運用コスト
- **配信タイミング**: いつ何を配信するか
- **コンテンツ削除・修正のフロー**: 報告対応、訂正対応

## SaaS / ツール系サービス

ユーザーがアカウントを作って継続利用するサービス。プロジェクト管理、コミュニケーションツール、業務効率化ツール。

### 検討すべき固有論点
- **料金プラン設計**: Free/Pro/Enterprise の機能差・価格
- **トライアル設計**: 無料期間・制限の設計
- **オンボーディング**: 初回ログイン後の導線
- **チーム機能**: 招待・権限・課金単位
- **データインポート・エクスポート**: 既存ツールからの移行
- **アカウント削除フロー**: データの扱い、復旧期間
- **使用量制限・課金トリガー**: API 回数、ストレージ容量

## EC / マーケットプレイス

商品を売り買いするサービス。

### 検討すべき固有論点
- **出品者・購入者の対応**: 二面性のあるサービス
- **決済フロー**: 支払い方法、エスクロー
- **配送・物流**: 配送業者連携、追跡
- **返品・返金ポリシー**: 期間、条件、処理フロー
- **手数料設計**: 売上に対する手数料率
- **本人確認**: KYC、銀行口座登録
- **トラブル対応**: 紛争解決、評価制度

## SNS / コミュニティ

ユーザー同士が交流するサービス。

### 検討すべき固有論点
- **モデレーション方針**: 何を許可し何を禁止するか
- **通報・ブロック機能**: 設計と運用フロー
- **おすすめ・タイムラインのアルゴリズム**: 表示順序の設計
- **通知設計**: 何を、いつ、どう通知するか
- **プライバシー設定**: 公開範囲の細かさ
- **アカウント停止・凍結のポリシー**: 段階的な対応
- **エコーチェンバー対策・健全性**: 健全な議論の促進

## エンタープライズ / 業務システム

社内利用、または企業向けの業務システム。

### 検討すべき固有論点
- **ロール・権限設計**: 詳細な権限階層
- **監査ログ**: 誰が何をいつ操作したか
- **SSO / SAML 連携**: 企業 ID 基盤との連携
- **データ保持・削除ポリシー**: 法令対応
- **オンプレ・専用環境**: 提供形態
- **SLA**: 稼働率保証
- **コンプライアンス**: GDPR、ISO、SOC2 など

## AI / LLM 活用サービス

LLM をコア機能として使うサービス。

### 検討すべき固有論点
- **モデル選定**: 速度 vs 精度 vs コスト
- **プロンプト管理**: バージョン管理、テスト
- **コスト管理**: トークン使用量の監視と制御
- **ハルシネーション対策**: 誤情報の検出と対応
- **ファインチューニング vs RAG**: 知識の与え方
- **ストリーミング設計**: UX 上の見え方
- **利用制限**: ユーザーあたりの使用量

## ハードウェア連携 / IoT

物理デバイスと連携するサービス。

### 検討すべき固有論点
- **デバイス管理**: 登録、ファームウェア更新
- **オフライン対応**: 接続切れ時の挙動
- **データ同期**: クラウドとデバイス間
- **電力管理**: バッテリー駆動の場合

## ゲーム / エンタメ

ゲームやエンタメ系サービス。

### 検討すべき固有論点
- **進捗・セーブデータ管理**: クラウド同期
- **マッチメイキング**: マルチプレイ
- **チート対策**: 不正検知
- **課金アイテム設計**: ガチャ、シーズンパス
- **コミュニティ機能**: ギルド、フレンド

## モバイルアプリ特有の論点

ネイティブモバイルアプリがある場合の追加論点。

### 検討すべき固有論点
- **iOS/Android の差分**: プラットフォーム固有の機能・制約
- **アプリストアの審査対応**: 拒否されやすいポイント
- **プッシュ通知設計**: 通知許可、頻度、内容
- **アプリ内課金**: ストア手数料 vs Web 課金
- **アプリ間連携**: Universal Links、Deep Links
- **オフライン対応**: ネット切断時の挙動

## ブラウザ拡張機能特有の論点

Chrome 拡張機能などがある場合の追加論点。

### 検討すべき固有論点
- **Manifest V2 / V3**: 仕様の制約
- **権限設計**: 過剰な権限要求を避ける
- **ストア審査対応**: Chrome Web Store のレビュー
- **クロスブラウザ対応**: Firefox、Edge、Safari
- **Web アプリとの認証連携**: Cookie / Token

---

## 提案の仕方

サービス固有論点を察知したら、以下のように切り出す:

> 「○○ について、これまで [複数の選択肢 / 数値 / 判断] が出てきましたね。これはこのサービス固有の重要な論点なので、独立したノートとして残しておくと後で参照しやすいです。ノート化しますか?」

ユーザーの返答を待つ。

- YES → ノートを作成
- NO → 作らない（押し付けない）
- 「もう少し考えてから」→ 後でまた提案する余地を残す
