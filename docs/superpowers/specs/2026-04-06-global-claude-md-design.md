# グローバル CLAUDE.md デザイン

## 概要

グローバルな `~/.claude/CLAUDE.md` を作成し、Claude Code の全セッション・全プロジェクトに適用される共通ルールを定義する。

## 背景

- 参考: [neko-neko/dotfiles CLAUDE.md](https://github.com/neko-neko/dotfiles/blob/c447d8870e814ec8bdf10574f82aad28878f99e9/claude/CLAUDE.md)
- 参考元から Verification Contract と Audit Gate を除外し、コードレビュー・セキュリティ・テスト・コミットメッセージのセクションを追加

## 設計方針

- **ルールベース型:** 箇条書きで明確なルールを羅列するスタイル
- **言語非依存:** グローバル CLAUDE.md に言語・フレームワーク固有のルールは含めない。それらはプロジェクトごとの CLAUDE.md に委ねる
- **拡張の余白:** スキル依存の記述は核となるルールを汎用的に書き、将来スキルを追加した際に連携できる形にする

## 配置

- ファイルパス: `~/.dotfiles/claude/CLAUDE.md`
- `setup.zsh` の既存ループ処理（25-31行目）により `~/.claude/CLAUDE.md` へ自動的にシンボリックリンクされる
- セットアップスクリプトの変更は不要

## セクション構成

### 1. コミュニケーション方針

- 技術的に誤った意見には根拠を示して反論する
- 曖昧な指示に対しては推測で進めず確認する
- 出力方針: 設計・レビュー・分析は完全な形で一度に出力する

### 2. コードレビュー

- 厳格なレビュー姿勢
- セキュリティ上の懸念は最優先で指摘

### 3. セキュリティ

- OWASP Top 10 の脆弱性検出と即座の報告・修正
- セキュリティ改善の積極的な提案（依頼がなくても）
- シークレット・認証情報の混入防止

### 4. 実装規律

- 初回の実装パスでバリデーションを含める
- コミット前に linter・型チェッカー・フォーマッター・テストを実行
- LSP 優先利用
- 成果物（コード・ドキュメント・設計）を提出する前にセルフレビューを行い、プレースホルダー・矛盾・曖昧さがないことを確認する

### 5. テスト

- Unit・Integration・E2E・VRT すべてを対象に積極的に実装
- ゴールデンテストを基本アプローチ
- プロジェクトの規約に従う

### 6. コミットメッセージ

- デフォルトは Conventional Commits
- プロジェクト固有の規約や既存の履歴がある場合はそちらを優先

### 7. マルチエージェント

- 明確なスコープと完了条件
- 出力の検証
- Research → Synthesis → Implementation → Verification の分業
- 調査結果の丸投げ禁止（オーケストレーターが統合）
- 自己完結プロンプト
- read-only 並列化、write-heavy 直列化
- continuation vs fresh context の使い分け

### 8. セッション管理

- 完了タスクの要約・整理を先回りで行う
- コンテキスト圧縮警告時に作業状態の保存を提案
- 中断・再開に備えた構造化された状態維持

### 9. 実装前検証

- 依存ライブラリのバージョンと既存コードの確認
- 前提条件の明示

### 10. フォーマッタ・リンタのスコープ

- 変更したファイルのみに適用
- git diff --name-only で対象特定

### 11. Document Dependency Check

- depends-on frontmatter に基づくドキュメント影響チェック
- ユーザー承認後に更新（自動更新禁止）

### 12. Knowledge Capture

- タスク完了時にプロジェクト固有の知見を auto memory に記録
- 記録件数 0〜3 件、重複禁止、記録後通知

## スコープ外

- 言語・フレームワーク固有のルール（プロジェクト CLAUDE.md で管理）
- Verification Contract / Audit Gate（採用しない）
- セットアップスクリプトの改修（既存処理で対応可能）
